This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev              # Start full stack: Hono API on :3001 + Vite on :5173 (proxies /api → :3001)
npm run build            # Vite production build (also regenerates routeTree.gen.ts via plugin)
npm run type-check       # tsc -b (separate from build)
npm test                 # Vitest (watch mode)
npx vitest run           # Vitest single run
npx vitest run api/index.test.ts  # Run a specific test file

# Database
npm run db:generate      # Generate Drizzle migration SQL from schema changes
npm run db:migrate       # Apply pending migrations to dev DB
npm run db:migrate:prod  # Apply pending migrations to prod DB (reads creds from 1Password)
npm run db:status        # Show migration status for dev
```

DB credentials are read from 1Password via `op read`. Items: `Private → Apunta → apunta-dev-db` and `apunta-prod-db`. Requires `op signin` before running DB scripts.

## Architecture

### Request flow
```
Browser → Vite (:5173) → proxy /api/* → Hono (:3001 dev / Vercel Edge prod)
                                             ↓
                                    Drizzle ORM → Neon (PostgreSQL)
```

**API** (`api/app.ts`) — plain Hono app, no auth. `api/index.ts` wraps it for Vercel Edge (`export const config = { runtime: "edge" }`). `api/serve.ts` is the local dev server (excluded from Vercel via `.vercelignore`). ESM imports in `api/` require explicit `.js` extensions.

**Frontend** — service/hook/view layered architecture:
- `src/services/` — raw `apiClient` calls, return typed data
- `src/hooks/` — TanStack Query wrappers with optimistic updates; views never call `apiClient` directly
- `src/routes/` — TanStack Router file-based routes; `routeTree.gen.ts` is auto-generated by the Vite plugin on every build/dev start, do not edit manually

### Key data flow patterns
- **Optimistic updates**: `cancelQueries → getQueryData (snapshot) → setQueryData (optimistic) → onError: restore snapshot → onSettled: setQueryData with server response`
- **Slug resolution**: Lists can be accessed by UUID or custom slug. All endpoints use `listWhere(param)` helper which guards UUID casting with a regex — passing a non-UUID string directly to `eq(lists.id, ...)` causes a Postgres type error.

### Database schema
Schema lives in `src/db/schema/lists.schema.ts`. The `index.ts` re-export uses **no `.js` extension** (drizzle-kit CJS incompatibility); the `client.ts` import uses `.js`. Migrations in `src/db/migrations/`.

### Testing
Tests use `app.request()` (Hono test helper) with a mocked `db` via `vi.mock('../src/db/client')`. The mock must be set up before importing `app`.
